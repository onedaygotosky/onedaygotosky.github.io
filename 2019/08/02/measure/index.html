view的绘制过程(一)——measure

Posted on 2019-08-02 In android

引言

Android中的整个UI体系中，对于客户端开发来说最重要的就是View类。所有诸如TextView、ImageView之类的常用系统控件都是继承自View，View是所有展示在屏幕上的元素的抽象，重要性不言而喻。

开发中，当碰到系统提供的控件不能满足展示需求的情况，需要实现自定义View；不光是自定义View这种比较高阶的应用，当界面比较复杂的时候，如果不能深刻理解整个绘制过程，实际显示效果可能跟想象大相径庭。

对绘制过程有过一定了解的一定都知道View的整个绘制过程可以分为measure、layout、draw三个步骤。作为系列文章的第一篇，本篇文章将讲解绘制过程的第一步，measure过程。

measure具体做什么

measure就是决定各个View具体应该多大的过程。知道了每个View应该多大，才能进行下面的layout布局操作和draw绘制到屏幕操作。一个View的大小受它父容器大小的影响，一个容器实际的大小也受它的子View实际大小的影响。

概念

首先先厘清几个概念。

  • View : 表示屏幕上的一块展示区域，位于整个用户界面系统的最底层，所有其他控件都集成自它

  • ViewGroup: 表示一个可以容纳View的容器，继承自View，所以他本身也是可绘制的

  • ViewParent: 表示可以是View的parent的接口，上面的ViewGroup就实现了这个接口；它的存在是为了View能跟上一层交互

view hierarchy

从上图可以看出来，整个View体系是层级组织的，不管整个体系中有多少复杂的控件，都可以分成容纳其他控件的容器和简单控件两种，构成整个层级每个view的绘制过程由上一级控制，不断往上传递。就像写Activity的时候通常会调用setContentView指定一个xml布局文件作为这个activity的布局。那么所有在这个页面上的view，往上追溯都会到contentView这一层。

不过，实际上Activity中view hierarchy最顶级不是contentView。一个Activity，也就是用户看到的界面，从逻辑层面来看，大概是这样：

这里主要是想先建立起对UI界面上view层级的一个感性的认识，不光是measure过程，还有layout过程和draw过程都是在这套层级体系下工作的。

Measure过程

measure过程简单来说，就是通过一定的算法得到最终每个view应该有的高宽。measure过程执行完后，就可以通过getMeasureHeight()和getMeasureWidth()获取测量过程得出的高宽。

MeasureSpec

上面看到的View的measure方法签名为public final void measure(int widthMeasureSpec, int heightMeasureSpec)，看起来这两个入参就是影响测量出来的view的大小的关键。

看下方法实现的代码就会发现，虽然这两个入参是int型的，但是频繁通过MeasureSpec类做诸如MeasureSpec.getSize()、MeasureSpec.getMode()的操作。看起来MeasureSpec在measure过程中有大作用，那么它究竟是何方神圣？

A MeasureSpec encapsulates the layout requirements passed from parent to child

简单说MeasureSpec就是封装了parent测量child时传递的数据。既然提到了封装，那么说明里面包括了不止一种信息，从方法名上就能看出来是测量模式(mode)和实际期望值(size)。

在一个int中存储多个数据目的是为了节省空间，也可以减少方法参数数量，一般实现就是把int的32位分成几段，再用mask分别取这几段的值。MeasureSpec实际也是这么做的。

具体来说，int在java中长度是32位(bit)，前两位用来表示mode，后30位用来表示size。

MeasureSpec的一大作用是定义了UNSPECIFIED、EXACTLY、AT_MOST三种mode，它们是测量过程中最重要的影响因素之一，看名字就大概知道分别表示了：未定义大小、具体的大小、最大不超过，配合size根据一定的规则就能算出最终的测量结果。具体的算法后面分析。

LayoutParams

决定measureSpec的规则

直接把代码贴出来，加上了详细的中文注释。虽然看起来比较长，但是逻辑很清晰易懂，加上的中文注释也说明了这样写的原因。

首先先明确：

  • 调用时传进来的是spec是父容器的measureSpec，childDimension一般情况下是layoutPrams.width/layoutPrams.height
  • 这个方法的作用是得到子View的measureSpec(最终影响子View的测量大小)
  • 最终结果受父容器的measureSpec和自身的layoutParams影响

那么，忽略掉padding，从一个被父容器测量的子view的视角来看，整个逻辑可以很清晰地用一个表格表示出来：

EXACTLY AT_MOST UNSPECIFIED
WRAP_CONTENT EXACTLY AT_MOST EXACTLY
MATCH_PARENT AT_MOST AT_MOST UNSPECIFIED
明确的大小(45px) EXACTLY EXACTLY UNSPECIFIED

一般各种ViewGroup都会遵循这个规则，但是如果应用不同的规则也是可以的。

总结

  • 整个用户看到的界面由View和ViewGroup组成，ViewGroup继承自View，而且能容纳View，所以形成了树状结构
  • 整个用户界面的绘制过程过程分为measure(决定各个View应该多大)，layout(实际放置各个View的位置)，draw(把View画到屏幕上)
  • view hierarchy逻辑上的根是ViewRootImpl，所有界面的改动最终都会一层一层往上通知到这里。performTraversals可以看做是整个绘制过程的起点。
  • ViewGroup在measure里会测量子View的大小，它的子View也可能是ViewGroup，这样就不断传播下去。
  • ViewRootImpl持有的mView就是容纳了整个屏幕的view(不包括系统的展示)，对应在Activity这个场景来说，它的measureSpec的mode为EXACTLY，size为屏幕大小，所以我们看到的是全屏的。
  • 对于一个View来说，它最终的measureSpec受父容器的measureSpec和自身的layoutParams共同影响。
